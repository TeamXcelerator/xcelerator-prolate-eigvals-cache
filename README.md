# xcelerator-prolate-eigvals-cache

Consolidated, public cache of **prolate-wave operator eigenvalue
spectra** — the eigenvalues of the CCM "educated-guess" operator PW_λ —
used across Team Xcelerator's high-precision spectral-triple work.

> **Author:** Ronnie Andrews, Jr. (Team Xcelerator Inc.®)
> **License:** See `LICENSE`. Public for academic verification and
> reproduction; redistribution and commercial use restricted.

## What this is

A prolate eigenvalue spectrum is uniquely determined by three integers:

- `lambda_sq` (λ²) — the CCM prime-cutoff parameter
- `ngrid` (N) — the finite-difference grid size (the operator is
  discretized on `N` interior nodes of `[-λ, λ]`, giving an `N×N`
  symmetric tridiagonal matrix)
- `prec` — the working precision, **in bits**

The spectrum is the full set of `N` eigenvalues of the prolate-wave
operator PW_λ (a symmetric tridiagonal Sturm-Liouville operator built
by 3-point finite differences on `[-λ, λ]`), computed in ascending
order. It is **project-agnostic**: for a given `(λ², N, prec)` it is
the same operator no matter which computation produced it. That
universality is why these caches can be consolidated into a single
shared repository.

This is a **distinct quantity** from the sibling caches:

- [`xcelerator-tau-cache`](https://github.com/TeamXcelerator/xcelerator-tau-cache)
  stores the Weil-form τ-matrix (a *different operator*).
- [`xcelerator-weil-eigvec-cache`](https://github.com/TeamXcelerator/xcelerator-weil-eigvec-cache)
  stores the Weil operator's smallest *eigenvector*; this repo stores
  the prolate operator's *eigenvalues* (a different operator *and* a
  different quantity).

The `ngrid` key (a finite-difference spatial resolution) is
deliberately distinct from the Weil caches' `nmodes` key (a Fourier
mode cutoff), so prolate filenames never collide with τ/ξ filenames.

Each spectrum is generated by the Xcelerator Toolkit
(`xc-spectral::prolate::hp`, via `tridiag_eigenvalues_hp`) and stored
as a deflated zip of the canonical JSON form.

## Why it exists

The prolate spectrum is expensive to produce: a full symmetric
tridiagonal QR at high precision. At HP-1000 (`prec=3338`) with
N=8001 this is roughly **30 minutes** of wall-time per config. The
output — a vector of `N` eigenvalues — is only a few MB. Recomputing
the spectrum on every fresh cloud run wasted large amounts of compute.
This repo lets any run **fetch only the spectra it needs** — by
deterministic URL, no clone required — turning that 30-minute QR into
a few-second download.

## Layout

Files are organized **precision-first, then by λ², then bucketed by
ngrid** in thousands — consistent with the sibling
[`xcelerator-tau-cache`](https://github.com/TeamXcelerator/xcelerator-tau-cache)
and
[`xcelerator-weil-eigvec-cache`](https://github.com/TeamXcelerator/xcelerator-weil-eigvec-cache)
repos. No directory exceeds GitHub's web-UI listing limit and the path
is deterministically computable from `(λ², N, prec)`:

```
prolate_eigvals_cache/
  prec{P}/
    lambda_sq{L}/
      ngrid{B}-{B+999}/
        lambda_sq{L}_ngrid{N}_prec{P}.json.zip
```

where:
- `P` = precision in bits (e.g. `3338` for HP-1000)
- `L` = λ² (the prime-cutoff parameter)
- `N` = finite-difference grid size (matrix dimension `N×N`)
- `B` = `floor(N / 1000) * 1000` (the lower edge of N's thousand-bucket)

**Example:** the HP-1000 (`prec=3338`) prolate spectrum for λ²=100,
N=4001 lives at

```
prolate_eigvals_cache/prec3338/lambda_sq100/ngrid4000-4999/lambda_sq100_ngrid4001_prec3338.json.zip
```

The **filename keeps the full `lambda_sq{L}_ngrid{N}_prec{P}` key**, so
each file is self-describing even if moved, and matches exactly what
the toolkit writes — no renaming needed on fetch.

### No byte-splitting

The eigenvalue vector is small (`N` decimal strings, ≲ a few MB
compressed even at HP-1000/N=8001), always well under GitHub's 100 MB
file limit. Every config is therefore a **single `.json.zip`** — there
is no `.partXX` tier, matching the GL-node and Weil-eigvec caches.

### Path formula (for tooling)

```
prolate_eigvals_cache/prec{P}/lambda_sq{L}/ngrid{floor(N/1000)*1000}-{floor(N/1000)*1000+999}/lambda_sq{L}_ngrid{N}_prec{P}.json.zip
```

## Fetching (no clone)

This repo is public, so any file is one raw URL, e.g.:

```bash
curl -sSL -O \
  https://raw.githubusercontent.com/TeamXcelerator/xcelerator-prolate-eigvals-cache/main/prolate_eigvals_cache/prec3338/lambda_sq100/ngrid4000-4999/lambda_sq100_ngrid4001_prec3338.json.zip
```

The Xcelerator Toolkit does this automatically on a local cache miss
under `CacheMode::DynamicFetch` (the default): it probes the single zip
by deterministic URL and treats a 404 as a definitive miss (falling
through to a fresh compute), while retrying 429/5xx with backoff. The
cache key is only active for **integer-valued λ²** (the paper configs
13/100/1000); a non-integer λ² bypasses the cache and always computes.

## File format

Each `.json.zip` is a deflate archive containing a single entry named
`lambda_sq{L}_ngrid{N}_prec{P}.json`: a JSON array of `N` decimal
strings (the prolate eigenvalues in ascending order) at the stated
precision. The precision is not stored inside the file — it is carried
by the `lambda_sq{L}_ngrid{N}_prec{P}` filename key and supplied by the
caller. The Xcelerator Toolkit reads/writes this format directly and
structurally validates it on load:

- eigenvalue count = `N`
- ascending order (`e[k] ≤ e[k+1]`)
- ground state ≈ `2π·λ²` (the asymptotic prediction for the prolate
  operator's lowest eigenvalue; the finite-difference discretization
  contributes a deviation of ≲ a few percent, far above any
  cache-corruption noise)
- no NaN/Inf entries

A file that fails validation is skipped with a stderr warning; the
toolkit falls through to a fresh compute. Public audit API:
`verify_prolate_eigvals_cache_dir`.
